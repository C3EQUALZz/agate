# Покрытие и ограничения

Эта страница честно описывает, **что Agate инспектирует сегодня**, а что
пересылает без инспекции — чтобы вы разворачивали его, не переоценивая защиту.
[Модель угроз](threat-model.md) описывает *целевую* модель безопасности; эта
страница — честное *текущее* состояние. Полный анализ и дорожная карта по
закрытию пробелов находятся в репозитории:
[`docs/design/security-coverage-roadmap.md`](https://github.com/C3EQUALZz/agate/blob/main/docs/design/security-coverage-roadmap.md).

!!! warning "Прочитайте это, прежде чем полагаться на Agate как на единственный барьер"
    Сейчас Agate обеспечивает авторизацию вызовов инструментов (по **имени** и
    по правилам запрета **аргументов**), редакцию текста по подстроке и
    fail-closed для битых событий. Ряд угроз из модели угроз пока не реализован
    (ниже) — в частности **результаты** инструментов, содержимое **состояния** и
    rate-лимиты. До выполнения оставшихся пунктов дорожной карты считайте Agate
    одним из слоёв защиты, а не полным «межсетевым экраном» для агента.

## Что инспектируется

| Событие / ввод | Что происходит | Может ли политика вмешаться? |
|---|---|---|
| `TOOL_CALL_*` (собранный) | буферизуется в полный вызов инструмента, затем оценивается | **Да** — разрешить/запретить по **имени** и по правилам запрета **аргументов** (`[[policy.tools.deny_arguments]]`) |
| `TEXT_MESSAGE_CONTENT` | сканируется на шаблоны секретов | **Да** — редактировать (literal, без учёта регистра, по подстроке) |
| Жизненный цикл (`RUN_*`, `STEP_*`) | порядок проверяется конечным автоматом `Run` | только структурно (без вердикта политики) |
| `STATE_SNAPSHOT` / `STATE_DELTA` | проверяется бюджет размера / числа операций | **Нет** — содержимое автоматически разрешается |
| «Нога» запроса (`RunAgentInput`) | авторизуется `tools[*].name`; текст `user`-сообщений проверяется на SSRF-URL | **Да** — отклонить до пересылки |
| Битые **известные** события | распознанный тип с отсутствующим/пустым обязательным полем нельзя инспектировать → обрабатывается по `[policy].on_malformed_event` (по умолчанию `terminate`) | **Да** — по умолчанию fail-closed |

## Что пересылается без инспекции

| Событие / ввод | Почему это важно |
|---|---|
| **Результаты** инструментов (`TOOL_CALL_RESULT`) | автоматически разрешаются — поверхность косвенной инъекции и эксфильтрации не инспектируется |
| Содержимое мутаций **состояния** | только бюджеты размера/числа операций; операции JSON Patch (RFC 6902) не валидируются |
| `RAW`, `CUSTOM`, `REASONING_ENCRYPTED_VALUE` | непрозрачные — пересылаются как есть, не инспектируются |
| Неизвестные / будущие типы событий AG-UI | пересылаются сырыми |
| Скрытые поля запроса (`system`, `forwardedProps`, `context`, входящий `state`) | не извлекаются, поэтому инъекция в них не проверяется |

## Операционные ограничения

- **Аутентификация** выключена по умолчанию (открытый прокси). Задайте
  `[proxy].api_keys` или поставьте перед Agate шлюз. См.
  [Конфигурацию](../getting-started/configuration.md).
- **Бюджеты против DoS** сегодня: глобальный лимит конкурентности, лимит размера
  тела запроса и таймауты подключения/чтения. **Нет** rate-лимита по
  ключу/сессии и **нет** ограничения размера/темпа потокового ответа клиенту.
- **Полнота аудита:** записи ставятся в очередь ограниченного outbox; при
  устойчивом противодавлении запись может быть отброшена (с логом и счётчиком)
  без остановки плоскости данных — поэтому переполненный outbox может оставить
  пробел в журнале прозрачности. Следите за метрикой отбрасывания аудита.
- **TLS** терминируется на прокси (необходимо для инспекции открытого текста);
  по умолчанию выключен, настраивается в секции `[tls]`.

## Дорожная карта

Закрытие оставшихся пробелов упорядочено в
[`security-coverage-roadmap.md`](https://github.com/C3EQUALZz/agate/blob/main/docs/design/security-coverage-roadmap.md):
вовлечение состояния и результатов инструментов в политику, бюджеты rate/вывода
и более богатый язык политик в TOML (regex/glob, структурные предикаты по
аргументам). (Fail-closed для битых известных событий и литеральные правила
запрета аргументов уже реализованы — см. `[policy].on_malformed_event` и
`[[policy.tools.deny_arguments]]`.)
